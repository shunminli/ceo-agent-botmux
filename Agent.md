# Project Agent Instructions

本文件定义本仓库的项目级 Agent 开发原则。用户最新明确要求优先；若本文件与更具体的任务说明、代码注释或既有设计文档冲突，以更具体且更严格的要求为准。

## 项目定位

本仓库维护一套基于 BotMux 的“一人公司”多 Agent 协作设计。默认由 CEO Agent 面向用户，负责目标澄清、任务拆解、Owner 分配、进度整合和最终交付；Tech Design、DevOps、Validation Agent 分别承担方案设计、工程落地和独立验证。

## 标准完整链路

每个开发任务默认按以下技能链路推进，除非用户明确要求只做更窄范围的动作：

1. `task-plan`：先判断目标、范围、成功标准、影响面、风险和验证方案。复杂任务、上线任务、高风险改动建议先使用它。
2. `do-and-done`：按计划开始实现，不只修用户点名的问题，也闭环当前影响面内看到的根因、相邻风险和验证缺口。
3. `say-and-do` / `say-and-replay`：中途按需使用。需要对齐目标和偏差时用 `say-and-do`；需要完成百分比和证据回放时用 `say-and-replay`。
4. `done-and-verify`：实现后用真实入口、真实依赖、真实数据或生产等价链路验证。单测、mock、本地 smoke 只能算前置证据。
5. `repo-local-memory`：如果代码语义、模块职责、功能行为或重要历史变化发生了变化，就更新本地 memory。只更新触碰范围，不写任务流水账。
6. `verify-and-summary`：最后做收尾，说明做了什么、验证了什么、还有什么风险或 TODO，并给出可上线、有条件可上线或不建议上线的判断。

简化理解：`task-plan` 负责想清楚，`do-and-done` 负责做到底，`done-and-verify` 负责证明真的成了，`repo-local-memory` 负责留下稳定知识，`verify-and-summary` 负责收口交付。`say-and-do` 和 `say-and-replay` 是中途对齐和进度解释工具。

## 协作边界

- CEO Agent 是默认对人接口；用户需求先转成目标、范围、Owner、验收标准和交付顺序。
- 涉及架构、平台、数据流、安全、成本或维护性的任务，先由 Tech Design Agent 给出方案和风险。
- 涉及研发、修复、部署、运维或工程质量的任务，由 DevOps Agent 落地。
- 重要方案、代码和交付物必须由 Validation Agent 做独立验证和质量门禁。
- 需求目标、验收标准、权限、凭证、安全、合规、不可逆操作、生产发布或关键取舍不清时，升级给 CEO Agent 或用户决策。

## 工程原则

- 优先遵循现有结构、命名、文档风格和本地 helper；不要为了单次任务引入不必要抽象。
- 修改范围要贴合请求和影响面，避免无关重构、格式化噪音和元数据 churn。
- 处理结构化数据时优先使用结构化 API 或解析器，不用脆弱的临时字符串拼接。
- 文件编辑保持 ASCII，除非文件已有明确的非 ASCII 语境或内容本身需要中文。
- 代码注释只解释不明显的设计意图或复杂逻辑，避免重复描述代码表面行为。
- 任何已有的未提交改动都视为用户改动；不得擅自回滚、覆盖或重置。

## 验证与交付

交付前应尽量运行与变更匹配的真实验证。若无法运行，应明确说明原因、影响和剩余风险。

每次交付至少包含：

- 做了什么。
- 影响了哪些文件、文档、系统或配置。
- 运行了哪些验证，结果是什么。
- 仍有哪些风险、假设、跳过项或待决策事项。
